# CSGClaw

> Your Personal AI Team

CSGClaw 是 OpenCSG 推出的多智能体协作平台。

它想解决的不是“怎么把一个 Agent 做得更万能”，而是另一个更实际的问题：

**当任务开始变复杂时，怎么让一组 AI 像一个团队一样协作起来，同时又足够轻、足够安全、足够容易启动。**

---

## 快速开始

macOS / Linux：

```bash
curl -fsSL https://raw.githubusercontent.com/OpenCSGs/csgclaw/main/scripts/install.sh | bash
```

安装完成后：

```bash
csgclaw onboard --base-url <url> --api-key <key> --model-id <model>
csgclaw serve
```

执行 `csgclaw serve` 后，CLI 会打印 IM 打开地址，例如：

```text
CSGClaw IM is available at: http://127.0.0.1:18080/
Open this URL in your browser after startup.
```

把这个地址复制到浏览器中即可进入 IM 工作区。

安装脚本会从 GitHub Releases 下载预编译二进制，并把它放到你的 `PATH` 目录中。目前只提供 macOS arm64 和 Linux amd64 的预编译版本。

源码编译：

```bash
export CGO_ENABLED=1
go mod download
(cd third_party/boxlite-go && BOXLITE_SDK_VERSION=v0.7.6 go run ./cmd/setup)
go build ./cmd/csgclaw
```

## 为什么要做 CSGClaw

单个 Agent 在很多场景里已经很好用，但只要进入真实项目，问题很快就会出现。

你会发现：

- 一个 Agent 要同时承担太多角色，代码、文档、调研、测试全堆在同一个上下文里
- 多 Agent 协作并不自然，创建、分工、跟进、同步都要人工盯着
- 为了把系统跑起来，往往要先配置一堆环境、容器和通信组件
- 你想要隔离和安全，但又不想为了本地使用引入过重的运维负担
- 你希望在浏览器里就能用，也希望以后能接飞书、微信，而不是被某个固定通道绑死

CSGClaw 的出发点很直接：

**把多智能体协作做成一个真正能开箱即用的产品，而不是一套需要用户自己拼装和维护的方案。**

---

## CSGClaw 是什么

CSGClaw 可以理解为一个面向个人和小团队的 AI 协作层。

它提供一位 Manager 和一组可分工的 Worker，让你不再直接和一堆独立 Agent 分别沟通，而是通过一个统一入口完成：

- 目标表达
- 任务拆解
- 角色分配
- 进度跟踪
- 结果汇总

整个模式更接近“带一个 AI 团队做事”，而不是“轮流操作几个聊天窗口”。

```text
┌──────────────────────────────────────────────────────┐
│                      CSGClaw                         │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │ Manager                                        │  │
│  │ 理解目标、拆解任务、协调 Worker                │  │
│  └────────────────────────────────────────────────┘  │
│             ↓                    ↓                  │
│      Worker Alice          Worker Bob               │
│       前端 / UI              后端 / 接口            │
│                                                      │
│      WebUI / 飞书 / 微信 / 其他接入通道             │
└──────────────────────────────────────────────────────┘
                    ↑
                 你来做决策
```

## CSGClaw 的设计取舍

CSGClaw 和 HiClaw 关注的是相近的问题，但实现路线不一样。

如果用一句话概括，CSGClaw 更在意的是：

**轻量、易启动、默认可用。**

### 1. 一键安装，尽量少配置

CSGClaw 从一开始就把“先跑起来”放在优先级前面。

我们不希望用户为了体验多智能体协作，先学习一整套基础设施，或者先补齐一堆外部依赖。理想状态是：

- 安装动作足够少
- 默认配置足够完整
- 启动路径足够短

先用起来，再按需扩展。

### 2. Manager 基于 PicoClaw

CSGClaw 的 Manager 基于 **PicoClaw** 构建，目标不是追求更重的编排层，而是降低成本：

- 启动更快
- 占用更低
- 更适合本地机器和日常常驻运行

对于个人开发者和小团队来说，这种轻量化比“功能最全但很重”的系统更重要。

### 3. Sandbox 基于 Boxlite

隔离很重要，但很多本地产品在这一步把事情做重了。

CSGClaw 选择 **Boxlite** 作为 Sandbox 基础设施，核心考虑是：

- 不需要用户安装和维护 Docker
- 依然保留必要的隔离能力
- 把安全边界做成默认能力，而不是额外门槛

这不是简单地“换一个底层组件”，而是在用户体验和安全之间做了更务实的平衡。

### 4. 默认 WebUI，不绑定单一 Channel

CSGClaw 不把某一种通信协议当作唯一入口。

默认情况下，你可以直接通过 WebUI 使用它；如果你的工作流已经在别的工具里，也可以继续接入其他 Channel，例如：

- 飞书
- 微信
- Matrix
- 其他可扩展消息入口

这使得 CSGClaw 更像一个平台，而不是某个特定消息系统的附属品。

## 它是怎么工作的

CSGClaw 的核心并不复杂，重点在于把关键角色划清楚：

### Manager

Manager 负责：

- 接收你的目标
- 判断任务是否需要拆解
- 选择合适的 Worker 来执行
- 跟踪当前进展
- 在关键节点把信息回收并汇总给你

它更像一个调度中枢，而不是一个什么都亲自做的超级 Agent。

### Worker

Worker 负责具体执行。

不同 Worker 可以面向不同职责存在，例如：

- 前端开发
- 后端开发
- 测试验收
- 文档撰写
- 信息调研

这样做的好处很直接：上下文更干净，角色更清晰，协作更容易组织。

### Sandbox

Worker 的执行环境由 Boxlite 提供隔离。

这意味着 CSGClaw 不需要把“安全”和“易用”当成二选一：

- 你不必先安装 Docker
- 系统也不必直接裸奔在宿主环境里

### Interface Layer

用户交互层默认由 WebUI 提供，同时保留向外接入其他 Channel 的能力。

这层设计的目标不是炫技，而是简单：

- 浏览器可以立即使用
- 团队沟通工具可以按需接入
- 平台本身不被某个入口锁死

## 一个典型场景

下面这个例子比概念更能说明 CSGClaw 想解决什么：

```text
你：做一个简单的产品官网，包含首页、登录页和后台雏形

Manager：收到，我拆一下任务
  - Alice 负责首页和登录页
  - Bob 负责后台接口和数据结构
  - Carol 负责联调和检查

Alice：开始处理页面结构和样式
Bob：开始定义接口和示例数据

Manager：当前前后端并行进行中，等第一版可联调结果

你：登录页需要支持 GitHub 登录

Manager：收到，已同步给 Alice 和 Bob

Carol：第一轮联调发现登录返回字段缺少用户头像

Manager：问题已记录，Bob 先修接口，Alice 等字段确认后更新展示
```

这里真正重要的不是“能不能创建多个 Agent”，而是：

**系统有没有把协作关系组织起来。**

## 为什么它更适合日常使用

很多多智能体系统在概念上都成立，但真正使用时容易卡在两个地方：

- 太重，跑起来麻烦
- 太散，用起来费劲

CSGClaw 想避开这两个问题。

### 更轻

- Manager 基于 PicoClaw
- Sandbox 基于 Boxlite
- 默认入口就是 WebUI

### 更快

- 启动链路更短
- 本地依赖更少
- 更适合作为常驻工具使用

### 更自然

- 你面对的是一个统一的协作入口
- Worker 可以按职责分工
- 通信通道可以根据实际场景接入

### 更务实

CSGClaw 不假设用户有完整的基础设施，不假设用户愿意维护 Docker，也不假设所有人都想把系统绑在同一套消息协议上。

它优先解决的是“多数人今天就能不能开始用”。

---

## 适合谁

CSGClaw 主要面向这些场景：

- 想把 AI 从单助手升级为协作团队的独立开发者
- 想降低多智能体使用门槛的小团队
- 更看重启动速度、资源占用和默认体验的用户
- 不想把入口绑定在 Matrix 上，而是希望直接从 WebUI 开始的人

---

## 产品方向

CSGClaw 的目标不是做一个更复杂的 Agent 平台，而是做一个更容易真正落地的 AI Team 平台。

“Your Personal AI Team” 不是一句包装语，它对应的是一组很具体的要求：

- 安装要足够简单
- 启动要足够迅速
- 协作要足够清晰
- 隔离要足够实用
- 入口要足够灵活

如果一套多智能体系统不能同时满足这些条件，它就很难成为日常工具。

这就是 CSGClaw 想走的方向。
